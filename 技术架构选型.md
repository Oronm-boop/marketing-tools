# MDT AI Agent PC端桌面应用技术架构选型

## 1. 项目定位

本项目是一个运行在 PC 桌面的 AI 营销工具应用，基于现有 `electron-app` 脚手架继续开发。产品界面参考 `design.md` 与 `Agent UI参考.png`，核心功能聚焦在两个互斥工具：

- SEO 关键词生成
- 宣传文案生成

本次技术架构优先满足以下目标：

- 桌面端可安装、可打包、可后续升级。
- UI 开发效率高，能快速落地设计稿中的深色卡片、表单、工具切换器。
- AI 生成请求、用户配置、本地缓存等能力有清晰边界。
- 渲染进程与主进程职责分离，避免 Electron 常见安全风险。
- 尽量沿用现有脚手架依赖，减少无必要的技术迁移。

## 2. 总体选型结论

| 模块 | 选型 | 结论 |
| --- | --- | --- |
| 桌面框架 | Electron | 沿用现有脚手架，适合 PC 桌面应用与 Web UI 快速开发 |
| 构建工具 | electron-vite + Vite | 沿用现有配置，支持主进程、预加载进程、渲染进程分别构建 |
| 前端框架 | Vue 3 | 沿用现有脚手架，适合表单型工具界面 |
| 开发语言 | TypeScript | 全项目统一使用 TypeScript，提升 IPC、接口、表单数据可靠性 |
| UI 组件库 | Element Plus | 已在项目中引入，适合表单、按钮、弹窗、消息反馈 |
| 图标库 | lucide-vue-next | 建议新增，用于侧边栏、设置、生成、复制、下载等图标 |
| 状态管理 | Pinia | 已在项目中引入，用于工具状态、用户设置、生成结果状态 |
| 状态持久化 | pinia-plugin-persist | 已在项目中引入，仅用于非敏感 UI 状态 |
| 路由 | Vue Router | 已在项目中引入，用于首页、设置页、关于页等页面组织 |
| 网络请求 | Axios | 已在项目中引入，用于调用后端 API 或 AI 服务网关 |

| 代码质量 | ESLint + Prettier + vue-tsc | 沿用现有脚本，开发前后保持类型与格式检查 |

```text

│  └─ index.d.ts                # window.api 类型声明
│
└─ src/renderer/src
   ├─ App.vue
   ├─ views/
   │  ├─ Home.vue               # 主工具页
   │  ├─ Settings.vue           # 设置页
   │  └─ About.vue
   ├─ components/
   │  ├─ layout/                # 侧边栏、主布局、窗口内容布局
   │  ├─ marketing-tools/       # SEO/文案工具相关组件
   │  └─ common/                # 通用按钮、空状态、结果卡片等
   ├─ store/
   │  ├─ useToolStore.ts        # 当前工具、表单、生成状态
   │  ├─ useSettingsStore.ts    # 主题、接口地址等非敏感设置
   │  └─ useResultStore.ts      # 生成结果、历史记录索引
   ├─ api/
   │  ├─ client.ts              # Axios 实例
   │  └─ generation.ts          # AI 生成相关接口
   ├─ domain/
   │  ├─ seo.ts                 # SEO关键词 payload、结果类型、提示词模板
   │  └─ copywriting.ts         # 宣传文案 payload、结果类型、提示词模板
   ├─ router/
   ├─ utils/
   └─ assets/
```

## 4. 进程分层职责

### 4.1 主进程 `src/main`

主进程负责桌面应用能力，不直接承载 UI 业务。

- 创建和管理主窗口。
- 控制应用菜单、外链打开、窗口尺寸、最小化/关闭行为。
- 处理需要 Node.js 或系统权限的能力，例如文件读写、导入导出、本地配置、自动更新。
- 承接敏感能力，例如本地 AI Key 读取、系统凭据访问、代理请求。
- 注册 IPC handler，向渲染进程提供受控能力。

建议调整：

- `BrowserWindow` 显式设置 `contextIsolation: true`。
- 保持 `nodeIntegration: false`。
- 如无特殊依赖，优先开启 `sandbox: true`；若当前依赖不兼容，再保留 `sandbox: false` 并收紧 IPC 白名单。
- 禁止渲染进程直接访问 Node.js API。

### 4.2 预加载进程 `src/preload`

预加载进程是主进程与渲染进程之间的安全桥。

- 使用 `contextBridge.exposeInMainWorld` 暴露 `window.api`。
- 只暴露业务需要的方法，不暴露完整 `ipcRenderer`。
- 所有入参和返回值使用 TypeScript 类型约束。
- 对高风险能力增加参数校验。

推荐暴露示例：

```ts
window.api = {
  generateSeoKeywords(payload),
  generateCopywriting(payload),
  saveSettings(payload),
  readSettings(),
  exportResult(payload)
}
```

### 4.3 渲染进程 `src/renderer`

渲染进程负责用户界面和普通业务状态。

- 实现左侧品牌区、设置入口、右侧工具切换与动态表单。
- 使用 Element Plus 承载表单、按钮、弹窗、消息提醒。
- 使用 Pinia 管理当前工具、表单值、加载状态、生成结果。
- 使用 Axios 调用后端 API；如果涉及本地密钥或系统能力，则通过 `window.api` 走 IPC。
- 不直接读取本地文件系统，不直接保存敏感密钥。

## 5. AI 生成链路选型

AI 请求建议按安全要求分为两种模式。

### 5.1 推荐模式：后端 API / AI 网关

适合正式产品。

```text
Renderer 表单
  -> Axios 请求业务后端
  -> 后端保存密钥并调用大模型
  -> 返回结构化生成结果
  -> Renderer 展示、复制、导出
```

优点：

- API Key 不进入客户端包体，安全性更好。
- 可统一做限流、计费、日志、模型切换、提示词版本管理。
- 方便后续接入用户体系与套餐权限。

现有 `electron.vite.config.ts` 已配置 `/api` 代理，可以继续沿用该方向。

### 5.2 可选模式：本地配置 AI Key

适合内部工具或个人使用。

```text
Renderer 表单
  -> window.api.generate(...)
  -> Main Process 读取本地安全配置
  -> Main Process 调用 AI Provider
  -> 返回结果给 Renderer
```

注意：

- 不要把 AI Key 写死在前端代码或 `.env` 后打入渲染进程。
- 本地密钥建议使用系统凭据管理工具，例如 `keytar`。
- 非敏感设置可以使用 `electron-store` 或 Pinia 持久化。

## 6. UI 与交互技术方案

### 6.1 布局方案

根据 `design.md`，应用采用左右分栏布局：

- 左侧固定侧边栏：品牌标识、底部设置入口。
- 右侧主工作区：工具切换器、动态表单、生成按钮、结果展示区。
- 不保留会议入口、新建聊天、历史会话列表、语音输入控件。

### 6.2 组件方案

建议拆分为以下核心组件：

| 组件 | 职责 |
| --- | --- |
| `AppShell.vue` | 应用整体布局 |
| `Sidebar.vue` | 左侧品牌与设置入口 |
| `ToolSwitcher.vue` | SEO / 宣传文案互斥切换 |
| `SeoKeywordForm.vue` | SEO关键词生成表单 |
| `CopywritingForm.vue` | 宣传文案生成表单 |
| `GenerateButton.vue` | 统一生成按钮与加载态 |
| `ResultPanel.vue` | 生成结果展示、复制、导出 |
| `SettingsDialog.vue` | API地址、主题、偏好设置 |

### 6.3 样式方案

沿用 Less，并将设计规范抽为 CSS 变量：

```less
:root {
  --app-bg: #0b0b0f;
  --sidebar-bg: rgba(20, 20, 28, 0.7);
  --card-bg: #1a1a1f;
  --input-bg: #25252b;
  --border-subtle: #2f2f37;
  --text-main: #ffffff;
  --text-secondary: #9ca3af;
  --brand-start: #7c3aed;
  --brand-end: #a855f7;
}
```

Element Plus 负责基础组件能力，自定义 Less 负责产品视觉，不建议大面积覆盖组件内部 DOM，避免升级成本过高。

## 7. 状态与数据设计

### 7.1 Pinia Store 划分

| Store | 内容 |
| --- | --- |
| `useToolStore` | 当前工具类型、表单数据、校验状态 |
| `useGenerationStore` | 生成中状态、错误信息、当前结果 |
| `useSettingsStore` | 接口地址、主题、用户偏好等非敏感配置 |
| `useHistoryStore` | 本地生成历史索引、收藏、最近使用 |

### 7.2 本地存储选型

MVP 阶段推荐：

- Pinia 持久化：保存当前工具、主题、表单草稿等非敏感轻量数据。
- IndexedDB：保存生成历史、结果草稿、收藏记录。

后续升级条件：

- 如果历史数据量较大。
- 如果需要复杂筛选、全文检索、批量导出。
- 如果需要跨窗口共享更稳定的数据读写。

满足以上条件时，可引入 SQLite，建议在主进程封装数据访问，渲染进程通过 IPC 调用。

## 8. 表单校验与数据契约

建议新增 `zod`，统一描述表单和接口数据结构。

SEO 关键词生成 payload：

```ts
type SeoKeywordPayload = {
  business: string
  features: string
  keywordCount: number
  searchEngines: Array<'baidu' | '360' | 'bing'>
}
```

宣传文案生成 payload：

```ts
type CopywritingPayload = {
  keyword: string
  keywordRepeatCount: number
  articleCount: number
  platforms: Array<'xiaohongshu' | 'baidu' | 'bilibili' | 'douyin' | 'wechatVideo' | 'weibo'>
}
```

校验规则：

- 必填文本去除首尾空格后不能为空。
- 数字输入限制最小值与最大值。
- 多选项至少选择一个。
- 提交前在渲染进程先校验；进入主进程或后端 API 前再次校验。

## 9. 安全策略

Electron 桌面应用需要重点控制渲染进程权限。

- 渲染进程不启用 Node.js 能力。
- `preload` 只暴露白名单 API。
- IPC channel 使用固定命名，不允许从渲染进程传任意 channel。
- 所有 IPC 入参做类型校验。
- 外链统一通过 `shell.openExternal` 打开，并校验 URL 协议只允许 `https:`。
- AI Key、用户 Token 等敏感信息不进入 Pinia、localStorage、IndexedDB。
- 打包前不要把测试接口、调试日志、明文密钥带入生产包。

## 10. 打包与发布

沿用 `electron-builder`。

现有脚本：

```bash
npm run dev
npm run build
npm run build:win
npm run build:mac
npm run build:linux
```

Windows 优先建议：

- 安装包格式：NSIS。
- 应用图标：使用 `build/icon.ico`。
- 构建产物：输出到 `dist`。
- 内部测试阶段可先不做代码签名。
- 正式发布建议补充代码签名，减少系统安全拦截。

后续如需要自动更新，可引入：

- `electron-updater`
- 私有更新服务器或 GitHub Releases

## 13. 开发阶段建议

### 第一阶段：界面骨架

- 重构 `Home.vue` 为营销工具首页。
- 实现 `AppShell`、`Sidebar`、`ToolSwitcher`。
- 移除会议、新建聊天、会话列表、语音入口。
- 按 `design.md` 落地深色主题与基础设计 token。

### 第二阶段：核心表单

- 实现 SEO 关键词生成表单。
- 实现宣传文案生成表单。
- 使用 Pinia 管理当前工具与表单状态。
- 完成必填、多选、数字范围校验。

### 第三阶段：AI 生成链路

- 封装 `api/generation.ts`。
- 统一 loading、错误、重试、取消请求状态。
- 生成结果进入 `ResultPanel` 展示。
- 支持复制结果与基础导出。

### 第四阶段：设置与本地历史

- 增加设置页或设置弹窗。
- 保存接口地址、主题、默认平台等非敏感偏好。
- 根据实际需求接入 IndexedDB 或 SQLite。

### 第五阶段：打包与发布

- 调整窗口尺寸、标题、图标、应用 ID。
- 完成 Windows 打包。
- 做生产环境配置检查。
- 如正式分发，再补代码签名与自动更新。

## 14. 最终推荐方案

本项目不建议更换为 Tauri、React 或其他桌面框架。当前 `Electron + electron-vite + Vue3 + TypeScript` 已经能很好覆盖需求，且脚手架已经搭好，继续沿用可以最快进入业务开发。

最终架构建议为：

```text
Electron 桌面壳
  + Main Process 负责系统能力与安全边界
  + Preload 负责受控 IPC 桥接
  + Vue3 Renderer 负责营销工具 UI
  + Pinia 管理状态
  + Axios 调用后端 AI 网关
  + Element Plus + Less 实现产品界面
  + electron-builder 完成桌面打包
```

优先把架构重心放在「清晰的渲染层组件拆分」「安全的 AI 调用链路」「可持续扩展的本地数据层」三件事上，这样后续增加更多营销工具时，不需要推翻现有结构。
